# Development of Reports

All reports of the manager terminal are implemented as a DLL. The libraries of reports are located in the reports\\ directory of the manager terminal and are loaded during manager terminal launch. Each library can contain multiple reports. All reports from the MetaTrader Manager package are distributed with a source code, report projects are located in the reports\\ directory.

It is recommended to develop the libraries of reports using Microsoft Visual C++. Disable the "Exception handling" option in the project settings (in the 'Project\\Settings' menu the 'C/C++' tab, the 'C++ Language' category). For a proper operation of the library of reports, use the Multithreaded DLL run-time (menu "Project\\Settings", tab "C/C++", category "Code Generation", option "Use run-time library"). It is recommended to use optimization at the highest possible speed (menu "Project\\Settings", tab "C/C++", category "General", option "Optimizations").

In the development of reports, a careful check of the entire code is required. Any exception error will negatively affect the operation of the manager terminal because it loads the reports libraries into its address space.

## Description of Exported Functions and Data Structures

Each library of reports must export four functions that will be called by the manager terminal.

RepGetVersion — returns the version of the library of reports.

RepGetModuleInfo — requests the text description of the library of reports. The function returns the number of reports implemented in the library. When you launch the manager terminal, it loads all report libraries, request the description of each library and adds a pop-up submenu with implemented reports to the menu of each section.

RepGetGeneratorInfo — returns the description of a report by index from zero to the total number of reports returned by the RepGetModuleInfo function. If the report with the specified index is not found in the library, the function returns FALSE, otherwise returns TRUE. Each report is described by the RepGeneratorInfo structure:

The description of the report contains its index in a library, its name and type, implemented file formats and the default extension of the report file. The field "type" describes the type of the report, and is a bit mask of the following values:

-   REPORT\_TYPE\_ONLINE — a report on online users, added to the context menu of the "Online" section of the manager terminal.
-   REPORT\_TYPE\_USERS — a report on accounts, added to the context menu of the "Users" section of the manager terminal.
-   REPORT\_TYPE\_ORDERS — a report on open orders, added to the context menu of the "Orders" section of the manager terminal.
-   REPORT\_TYPE\_REPORTS — a report on closed trades requested from the server, added to the context menu of the "Reports" section of the manager terminal.
-   REPORT\_TYPE\_HISTORY — a report on daily reports requested from the server, added to the context menu of the "History" section of the manager terminal.
-   REPORT\_TYPE\_JOURNAL — a report on server logs, added to the context menu of the "Journal" section of the manager terminal.
-   REPORT\_TYPE\_RAW — the flag of an unprocessed ("raw") report that can be added to other flags of report types. This report is called when you click "Save As..." in the context menu of the section; reports without the REPORT\_TYPE\_RAW flag are added to the submenu of the section's context menu.

Also the ReportParams structure of the report contains the source data for the report. Current group settings (the "user\_groups" field) and symbol settings (the "symbols" field) are passed to the function when generating reports of all types. Also, during the generation of any report, the copy of the current account base (the "users" field) is passed to the report generation function.

The ReportParams structure contains the source data for the certain types of reports:

-   For reports of type REPORT\_TYPE\_ONLINE, the "buffer" field contains the list of online users (OnlineUserRecord).
-   For reports of type REPORT\_TYPE\_USERS the "users" field contains the list of accounts.
-   For reports of type REPORT\_TYPE\_ORDERS the "trades" field contains the list of open positions.
-   For reports of type REPORT\_TYPE\_REPORTS the "trades" field contains the list of closed positions, the "buffer" field contains the list of margin levels (MarginLevel).
-   For reports of type REPORT\_TYPE\_HISTORY the "buffer" field contains the list of daily reports (DailyReport).
-   For reports of type REPORT\_TYPE\_JOURNAL the "buffer" field contains the list of server logs (ServerLog).

The "fn\_progress" field of the report parameters contains a pointer to the function defined in the manager terminal. The report generation function should periodically call the function to indicate the report generation progress and to check the cancellation of report generation. A value predetermined by the terminal (field "custom"), the current report generation step, and the total number of report generation steps are passed as the function parameters. If the "fn\_progress" function returns zero, the report generation must be continued, otherwise it must be stopped. The report generation progress is displayed in the terminal in this case. A user can cancel the report generation by clicking the "Cancel" button.

The base class CReport is a fully-functional class that provides the automatic support for the list of reports implemented in the library, and the call or certain type report generation. Here are the short descriptions of the CReport methods and data members:

-   RepGeneratorInfo m\_info — report description.
-   FILE \*m\_file — a pointer to the fie of the generated report.
-   CReport \*m\_next — a pointer to the next report in the list of report implemented in the library.
-   CReport — class constructor, fills the report description and adds the created report to the top of the list of reports in the library.
-   ~CReport — class destructor, deletes reports from the library.
-   Report — calling report generation by index in the HTML or CSV format.
-   GenerateHTML — a pure virtual method of a HTML report generation, which must be defined in the implementation of a certain report derived from CReport.
-   GenerateCSV — a pure virtual method of a CSV report generation, which must be defined in the implementation of a certain report derived from CReport.
-   GetInfo — gets a report description by index.
-   Total — gets the total number of reports implemented in the library.
-   Startup — an additional method that checks the report parameters and opens the file of the generated report.
-   WriteHeader — an additional method that writes to a HTML file the header and styles required for a correct display of values when the report is opened in Microsoft Excel.

The CReportClosedTrades class constructor simply calls the constructor of the base class. A CSV or HTML report is generated in the GenerateCSV and GenerateHTML functions. The generation of each report consists of several steps:

-   Checking the report parameters and opening of report file - performed in the Startup method of the base CReport class.
-   For a report in the HTML format, the output of the HTML file title and various styles - performed in the WhiteHeader method of the base CReport class.
-   Output of the report name, the name of the set of accounts, the report period, and the report table header.
-   Iteration through all closed positions and output of the report table rows.
-   Setting the background color of the row.
-   Selecting a class when outputting the symbol price.
-   Output of the report line.
-   Output of profits and losses.
-   Calculating total volumes and profits.
-   Periodic call of the manager terminal function to indicate the generation progress and to check the termination of generation.
-   Output of the final line of total volumes and profits.
