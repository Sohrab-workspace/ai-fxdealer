# Development of Data Feeds

The data feed structure provides the reenterability - i.e. one and the same data feed can be loaded multiple times and enable access to different data sources.

It is recommended to develop data feeds using Microsoft Visual C++. Disable the "Exception handling" option in the project settings (in the 'Project\\Settings' menu, the 'C/C++' tab, the 'C++ Language' category), because the server completely intercepts error handling and adds them to the crash log. It is also recommended to add version information as a resource.

For a proper operation of the data feed, use the Multithreaded run-time library (menu "Project\\Settings", tab "C/C++", category "Code Generation", option "Use run-time library"). It is recommended to use optimization at the highest possible speed (menu "Project\\Settings", tab "C/C++", category "General", option "Optimizations").

Naturally, the development requires thorough proofreading of the entire code, because any exceptional mistake will have a negative impact on the server performance, since it loads the data feed into its address space.

## Data Feed Development Features

The data feed is one of the major components of the server. Therefore very strict code security rules should be applied to the data feed. Any data received from the provider server should be carefully checked. Note that the MetaTrader server must be able to stop the operation of the data feed at any time. In this case, the data feed should to provide the fastest possible response to this operation.

All data feeds must be controlled. For example, some data feds may use APIs instead of sockets. The APIs cannot be stopped by a simple call of a function such as closesocket. In such cases, the data feed should create a control thread. One of the options is to check the last data feed activity. This check can be implemented by instructing the data feed to set the time of some variable from time to time, while the control thread would read and analyze whether its value exceeds a preset limit.

Using external libraries is not recommended. If any interface is provided to you as a DLL or in ActiveX, a better and a more secure solution is to develop a separate application, which would use these components, and to open a local server, to which the securely written data feed would connect.

Some provider servers feed news in the html format. If you need to represent a newsletter in the html format, you should use tags <html>, </html> before and after its body, otherwise the newsletter will be identified as a text.

When using the HTTP protocol as a method of access to the provider server, sometimes you may need to disable the transparent proxy, which some Internet providers may enable, as this may cause data feed malfunction.

This allows you to obtain the version of the data feed, its name, and information about the data feed author. It is important to specify the server, which the data feed uses for connecting to the data provider. You can also specify the default login information (login and password) and the data feed description, which may contain additional information about the data feed and its specific features.

One of the most important fields of this structure is FeedDescription::modes, which describes the type of data feed and possible interaction with it. It can have one of the following four values:

-   modeOnlyQuotes - the data feed can only receive quotes.
-   modeOnlyNews - the data feed can only receive news.
-   modeQuotesAndNews - the data feed can simultaneously receive quotes and news.
-   modeQuotesOrNews - the data feed can either receive news or quotes at a moment of time.

At the same time, the first three values ​​are used by the server to indicate the current data feed operation mode.

[DsCreate](/en/docs/mt4/api/datafeed_api/datafeed_api_exported_function/datafeed_api_dscreate) passes a pointer to the newly created abstract class [CFeedInterface](/en/docs/mt4/api/datafeed_api/datafeed_api_cfeedinterface), i.e. to a new data source. When a new source is created, it is added to the list of data sources. So, anyone who has created a data source, only works with it. This provides reenterability of the data feed.

[DsDestroy](/en/docs/mt4/api/datafeed_api/datafeed_api_exported_function/datafeed_api_dsdestroy) destroys a data source object that was earlier created using DsCreate, and removes it from the list.

The abstract class CFeedInterface provides a uniform operation of the server with any data feed regardless of its internal implementation. The server calls methods of the CFeedInterface class in a definite order, while the internal data feed implementation provides proper processing.

# The CFeedInterface Class

Let us view each of the virtual methods of the CFeedInterface class.

[Connect](/en/docs/mt4/api/datafeed_api/datafeed_api_cfeedinterface/cfeedinterface_connect) - before starting to receive quotes or news from a data feed, the server will call this function to enable the data feed to connect to the provider server. Parameters:

-   server - the ip address or the domain name of the server and the port. String format: "www.metaquotes.ru:443", "66.235.184.162:443". In some cases port number can be omitted. In this case the data feed is to use a default port.
-   login - login for authenticating.
-   password - password for authenticating.

Any of these parameters can contain an empty string. For example, if login and password are empty, then it is expected that no authentication is required for connecting to a provider server. In some cases the server parameter can also be empty, for example a data feed can read data stream from a file or in some other way.

[Close](/en/docs/mt4/api/datafeed_api/datafeed_api_cfeedinterface/cfeedinterface_close) - called when stopping a feed is required, this may be caused by server shutdown or by some other reason. Usually, when this function is called, a data feed closes connection to a server, or performs other deinitializing actions.

[SetSymbols](/en/docs/mt4/api/datafeed_api/datafeed_api_cfeedinterface/cfeedinterface_setsymbols) - in addition to calling the Connect function, in case of further request of quotes, the server initializes the data source with a recommended list of symbols. A string of symbols separated by a semicolon is received as the only parameter "symbols", example: "EURUSD;EURCHF;USDCHF;USDJPY".

A data feed can ignore this list of symbols. The sole purpose of this function is to minimize the traffic between the feeder and the provider server by requesting only the required symbols. However, there are feeders that cannot control the stream of symbols received from the provider server.

[Read](/en/docs/mt4/api/datafeed_api/datafeed_api_cfeedinterface/cfeedinterface_read) - the most used function in the data feed operation, it is responsible for the streaming of quotes and news. The only parameter of the function is a pointer to the data structure [FeedData](/en/docs/mt4/api/reference_structures/structure_feed/feeddata), which received information about quotes and news. A data feed can transmit to the server no more than 32 quotes at a time. You should fill each of the [FeedTick](/en/docs/mt4/api/reference_structures/structure_feed/feedtick) structures containing the bid/ask of a quote, the symbol, time and possibly the quote source. The number of transmitted quotes should be specified in the "tick\_count" field of the FeedData structure.

In contrast to the quotes, a data feed can only transmit one newsletter at a time. In this case you should specify the time of the new "news\_time" in the format of "2004.10.22 18:30". After it is necessary to fill in three important fields: the news "subject", the news "body", and "body\_length". Remember, that if the "subject" is empty, the server will skip this news. However, it is allowed to pass news with an empty body, in such a case it is assumed that the news only has a subject (no body).

Remember that the memory area pointed to by the "body" field is allocated by the server, it cannot be released, and you cannot reset the body pointer.

There are some auxiliary fields, including news "category" and "keywords", which can later be used for internal purposes, e.g. in the server API.

Sometimes, errors may occur in the data feed operation, such as connection failure or an error processing the provider server protocol. For such cases more informative fields such as "result" and "result\_string" are used, these fields will be displayed in the server logs.

[Journal](/en/docs/mt4/api/datafeed_api/datafeed_api_cfeedinterface/cfeedinterface_journal) - a service function, which is not connected with the data feeding. It is used for passing the data feed log to a server. The Journal function receives one parameter "buffer", which is a pointer to a 8 Kb area of memory, to which recent data feed logs should be passed.

Each of the virtual functions such as Connect, SetSymbols, Read, Journal has a mandatory implementation. In addition, there are two auxiliary Next functions, which are used for access to a single and protected class member "m\_next", which is a pointer to the next object of the CSourceInterface class. Next functions allow to search for and add data feed sources to the list of objects.

In addition to standard functions, the CSourceInterface class uses internal functions. The Login function implements an authentication procedure. ReadNews is designed in order to simplify the code of the Read function, which reads both quotes and news.

Functions IsReadible, SendString, ReadString, ReadCheckString serve to simplify work with the socket - checking socket readability, reading and sending strings.

This function has three pointer as input parameters, each of which needs to be checked. Then, for further use, we save them in the appropriate class members by using the "COPY\_STR" macro. The "COPY\_STR" macro implements secure string copying (described in StdAfx.h, see the example).

Since the 'server; parameter arrives in the form of a string containing the server address, and possibly the port, we will try to define it. If the connection port is not specified, the port whose value is initialized in the local variable "port" will be used.

The 'server' parameter can contain the ip address of eh server or its domain name. That is why, an attempt is made to define the ip address using "inet\_addr", and then, if an error is returned, the ip is determined based on the domain name using "gethostbyname".

Now, the data feed can try to connect to the server.

First the "inf" input is checked, which is a pointer to the "FeedData" structure containing data about quotes and news.

If connection is not established, an attempt is made to connect to the server. Next, if the "m\_loginflag" has not been set, the authentication procedure Login is called. In some data feeds this step is skipped. For example, if no authentication is required on the provider server. Then the data feed switches to the mode of waiting for the incoming data stream.

Some data feeds, if required by the provider server protocol, periodically send out a ping-message at a specified time interval.

Under the protocol used by UniFeeder, strings separated by newline character are received. After receiving the string, the data feed parses it, and depending on that takes a decision. For example, if the data feed receives a string that contains the substring "< News" at the beginning, it will pass control to the ReadNews function, which continues to receive and analyze the current news.

If a string of type "USDJPY 1.0106 1.0099" is received, the data feed will receive the bid/ask value for the appropriate symbol, and will fill the "FeedData" structure using this data.

Only one quote can be received at a time, so UniFeeder marks the "ticks\_count" field of the "FeedData" structure with the digit 1, i.e. it sets the number of quotes to 1. However, some data feeds are able to receive multiple quotes in one data packet and, consequently, they can transmit to the server multiple quotes, but no more than 32.

## UniFeeder Protocol

The below description of the UniFeeder protocol operation will help you understand its principles.

-   The data feed establishes TCP connections to a data source.
-   The data feed reads data line by line, the line separator is a sequence of "\\r\\n".
-   Authorization:
    -   After connecting, the data feed waits for a login request from the data source. A string containing "Login: " is expected. For example, "Login :\\n\\r". When the data feed waits for the login request, the following entry is added to the journal "waiting login request".
    -   After receiving the login request, the following message is added to the log: "login request received". Then the data feed sends a string with the login to the data source, e.g. "user\_test\\r\\n".
    -   The data feed waits for a password request from the data source. A string containing "Password: " is expected. When the data feed waits for the password request, the following entry is added to the journal: "login sent \[%S\], waiting password request".
    -   After receiving the password request, the following message is added to the log: "password request received". Then the data feed sends a string with the password to the data source, e.g. "password\\r\\n".
    -   The data feed waits for a notification of successful authentication. A string containing "Access granted" is expected. While the data feed waits, the following entry is added to the journal: "password sent, waiting access confirmation"
    -   After receiving the notification of authorization, the following log is added "Login: '% S' successful".
    -   The data feed sends a string with symbols, for which the data feed will be able to send quotes of the type "> Symbols:USDRUB,EURUSD,EURRUB\\r\\n".
    -   The data feed proceeds to the process of receiving data (quotes and news).
-   Receiving data:
    -   The data feed reads data line by line, the line separator is a sequence of "\\r\\n".
    -   Strings starting with the symbol ">" or "<" are skipped. The only exception is lines starting with "< News".
    -   If "< News\\r\\n" is received, the data feed switches to the process of news receiving:
        -   After "< News\\r\\n", the data feed reads the [NewsTopic](/en/docs/mt4/api/reference_structures/structure_news_mail/newstopic) structure.
        -   Then the data feed reads the news body. The body size is specified in bytes in the field NewsTopic::len.
        -   After reading the body, the news is sent to the platform, and the data feed returns to the data reception process.
    -   If a string does not contain "<" or ">", the string is considered to be a tick.
        -   String format: "<symbol name> <bid> <ask>\\r\\n". For example, "USDJPY 1.0106 1.0099\\r\\n".
        -   If bid or ask below zero, or bid id greater than ask, this tick will be skipped, and the following message will be added to the journal "failed to parse tick, invalid bid/ask".
